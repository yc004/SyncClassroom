# 故障排查

本节收录当前最常见的环境、启动和联调问题。

## 根目录无法启动指定端

先确认子模块已经初始化：

```bash
npm run repos:init
```

如果仍然失败，检查对应目录是否存在：

- `repos/core`
- `repos/teacher`
- `repos/student`
- `repos/editor-plugin`

## 教师端能启动，但学生端无法连接

优先检查三项：

1. 教师端服务是否已启动
2. 学生端输入的教师机地址是否正确
3. 教师机端口是否被防火墙或本地安全软件拦截

当前学生端默认连接教师端服务，而不是直接连接独立 core 服务。

## 教师端能看到课件，学生端不同步

重点检查运行时事件链路：

1. 教师端是否已经完成选课
2. 教师端服务是否正常广播 `select-course` 和 `sync-slide`
3. 学生端是否已经连接到正确的教师机地址

如果是联调环境，优先确认教师服务日志里是否已经收到学生连接。

## 课件资源加载失败

当前资源地址由教师端托管并下发。出现加载失败时，优先检查：

1. 课程资源文件是否仍在教师端课程目录中
2. 教师端课程扫描结果是否正常
3. 资源 URL 是否指向教师端服务地址

不要再假设资源由 core 提供。

## 提交文件没有保存

提交数据的持久化责任在教师端。应检查：

1. 教师端提交目录是否存在
2. 教师端是否具备目标目录写权限
3. 提交配置是否正确
4. 教师端服务日志中是否出现保存失败信息

## 修改子仓代码后，根仓没有变化

这是正常现象。子仓库和根仓是两层 Git：

- 子仓内提交，更新的是子仓自身历史
- 根仓只有在子模块指针变化后才会出现变更

如果要让根仓记录最新子模块引用，需要再执行一次：

```bash
git add repos
git commit -m "chore: update submodule pointers"
git push
```

## 文档里看到旧路径

当前推荐文档入口只有一个：

- [docs/README.md](../README.md)

如果看到 `docs/dev`、`docs/user`、`docs/components`、`docs/interaction` 这类旧路径，它们属于历史结构，应以新编号章节为准。
